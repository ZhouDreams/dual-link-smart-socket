## 二、跨模块模式（维度 E 汇总）

以下 8 个模式横跨多个模块，是系统性问题而非孤立缺陷。修复时应统一处理而非逐模块打补丁。

### 模式 1：destroy/stop 失败时泄漏资源（4 模块，系统性）

`destroy()` 调用 `stop()`，若 `stop()` 失败则直接 return 不释放 `me`/mutex，违反 `err.md` §3.2 "destroy 必须支持传入 NULL 并安全返回"的精神（虽 §3.2 字面只覆盖 NULL，但 destroy 的核心契约是"总是释放资源"）。

| 模块 | 文件:行号 | 问题 |
|------|-----------|------|
| app_controller | `app_controller.c:255-273` (M1) | stop 失败 → me + mutex 泄漏 |
| network_manager | `network_manager.c:391-393` (D-1) | stop/clear_link_rx_cb 失败 → 不调 free_resources，manager 泄漏且 `destroying=true` 导致系统不可恢复 |
| metering_service | `metering_service.c:198-201` (MS-03) | stop 失败 → mutex + 对象泄漏 |
| button | `button.c:258-262` | iot_button_delete 失败 → me + mutex 泄漏 |

**关联**：metering_service `MS-02`（`metering_service.c:363-364`）是变体——stop 内部 mutex take 失败时 `stopping` 标志永久为 true，服务永久锁定。

**建议**：统一 destroy 模式——stop 失败时记录 warning 但继续释放资源（删 mutex、free 对象），返回 stop 的错误码但不影响回收。

### 模式 2：esp_event_post 失败处理不完整（3 模块，含安全影响）

`esp_event_post` 可能因事件队列满而失败（ESP-IDF 默认 10ms 超时）。当前多处仅记日志就继续，导致状态不一致。

| 模块 | 文件:行号 | 后果 |
|------|-----------|------|
| metering_service | `metering_service.c:700-712` (MS-01) | **post 失败 → pending 电能增量未回滚 → 下一次 prepare 返回 `ESP_ERR_INVALID_STATE` → 服务静默卡死 → safety_guard 失去输入数据源** |
| safety_guard | `safety_guard.c:520-534` | post 失败 → DANGER/RELAY_OFF 快照被丢弃 → 继电器断开延迟最多 1s |
| relay | `relay.c:185,191-193` | post 在释放 mutex 后执行 → TOCTOU 竞态（事件乱序） |

**安全影响**：MS-01 最严重——metering_service 卡死后，safety_guard 不再收到快照，过流/过功率检测完全失效。这是本次审查中**最接近 🔴 的发现**，因 `persistence_samples=3` 的 3s 容错窗口使其未达 🔴 阈值。

**建议**：
- metering_service：post 失败后重新获取锁并回滚 pending 状态
- safety_guard：post 失败后保留 latest 快照供 `get_latest()` 读取
- relay：将 `esp_event_post` 移入 mutex 临界区（`esp_event_post` 异步拷贝数据，持锁不会死锁）

### 模式 4：handler 注册顺序影响安全路径时延（app_controller，安全相关）

- `app_controller.c:697-747` (M3)：`METERING_EVENT_SNAPSHOT` handler 注册早于 `safety_guard_start`
- esp_event 按注册顺序派发 → `app_controller_on_metering_snapshot`（含 `thingsboard_client_publish_telemetry` → JSON 格式化 + `calloc` + `network_manager_publish`）先于 `safety_guard_on_metering_snapshot` 执行
- LTE 链路下 publish 可能阻塞数十至数百毫秒，延迟 safety_guard 的规则判定和 `relay_set(SAFETY, false)`
- 违反 `architecture.md` §6.4 "安全保护必须是本地闭环"

**缓解因素**：safety_guard `persistence_samples=3`（3s 容错），实际延迟占比小。

**建议**：将遥测 publish defer 到独立任务/工作队列；或调整 handler 注册顺序使 safety_guard 先注册。

### 模式 5：分层契约违反（app_controller → thingsboard_client）

- `app_controller_internal.c:17` (M6)：应用编排层 include 了 `thingsboard_client_internal.h`（业务服务层内部头文件），调用 `tb_internal_format_power_limit_response`
- `architecture.md` §8.1 允许 app_controller 依赖"所有公共模块 API"，但不包括内部头文件

**建议**：将 power limit response 格式化提升为 thingsboard_client 公共 API。

### 模式 6：destroy 中无超时轮询（wifi_link + lte_link）

| 模块 | 文件:行号 | 问题 |
|------|-----------|------|
| wifi_link | `wifi_link.c` `wait_runtime_actions_drained` | 无超时上限的轮询 |
| lte_link | `lte_link.c:475-490` `wait_rx_callbacks_drained` | `while(true)` 轮询，rx_cb 阻塞则 destroy 永久挂起 |

**建议**：添加超时上限，超时后记录 warning 并强制继续释放（接受回调可能 use-after-free 的风险，或用 atomic flag 标记回调正在执行）。

### 模式 7：lifecycle API 未接线（系统级）

`rg` 验证以下 destroy 函数在整个 `main/` 下**从未被调用**：
- `network_link_destroy`
- `network_manager_destroy`
- `relay_destroy`（classes.md §3.6 提到但无调用方）

模块对象在程序生命周期内不显式释放。destroy 路径中的清理逻辑（回调注销、任务停止、mutex 删除）是死代码——未被执行过，可能隐藏未暴露的 bug。

**影响**：当前单进程嵌入式系统下可接受（程序退出即断电），但：
- destroy 路径的 bug 不会在上机时暴露
- 单元测试无法验证清理逻辑
- 未来若需热重启/模块替换，destroy 路径必须可靠

**建议**：至少在 host-side 测试中调用 destroy 验证清理逻辑；或接受当前设计但在文档中明确"对象生命周期 = 程序生命周期"。

### 模式 8：回调上下文安全性（多模块）

多个回调在非调用方任务上下文执行，部分使用 `portMAX_DELAY` 获取 mutex：

| 回调 | 执行上下文 | 风险 |
|------|-----------|------|
| button 回调 | esp_timer 任务 | app_controller 用 `portMAX_DELAY` 取 mutex，可能阻塞 esp_timer |
| wifi_link MQTT 事件 | esp_mqtt 内部任务 | 持锁调用可能阻塞的 API |
| lte_link lwlte 事件 | esp-lwlte 内部任务 | 同上 |
| esp_event handler | 默认事件循环任务 | metering/safety/relay handler 均 `portMAX_DELAY` 取 mutex |

**当前风险低**（mutex 持有者均为快速操作），但 `portMAX_DELAY` 意味着无上限等待——若发生优先级反转或下游阻塞，事件循环任务会无限阻塞，导致所有模块事件处理停滞。

**建议**：关键路径回调改用有限超时（如 `pdMS_TO_TICKS(100)`），超时后跳过并记录 warning。

## 三、Top 已确认发现（按影响排序）

以下为本次审查中影响最大的 10 条已确认发现（均为 🟡 中严重度，因无 🔴）：

### 资源泄漏

4. **`network_manager.c:391-393` (D-1)** — destroy 在 stop 失败时不调 free_resources → manager 泄漏且 `destroying=true` 导致系统不可恢复

6. **`lvgl_dashboard.c:830-837` (M1)** — `tick_ctx` 在 destroy 中置 NULL 不释放 → 每次 create→destroy 泄漏约 20 字节（根因：用泄漏规避 `esp_timer_stop` 不等待 in-flight 回调的 UAF 竞态）

### 并发竞态

7. **`lte_link.c:456`** — `set_active_impl` 无锁读取 mutex 保护字段 `destroying`/`lwlte` → 与并发 `destroy_impl` 存在 TOCTOU 竞态（潜在 use-after-free；当前因 destroy 未被调用而未触发）
8. **`relay.c:185,191-193`** — `relay_set`/`relay_toggle` 在释放 mutex 后才发布状态变化事件 → TOCTOU 竞态，多任务并发可导致事件乱序
9. **`bl0942.c:1071-1107`** — `bl0942_hard_reset` 持有 mutex 期间调用 `bl0942_power_cycle`（两次 `vTaskDelay` 共 2s）→ 阻塞并发 `bl0942_read`/`get_latest`，power_cycle 不访问 `me->` 任何字段，持锁无技术必要

### 分层/文档

10. **`app_controller_internal.c:17` (M6)** — 应用编排层 include 业务服务层内部头文件 `thingsboard_client_internal.h` → 违反 `architecture.md` §8.1 分层契约

## 六、审查方法说明

本次审查采用并行 subagent 方式：14 个模块分派给 13 个 general-type subagent（board_pinmap + relay 合并为一个 agent），每个 agent 独立完成阶段 2（审查报告）+ 阶段 3（验证报告），不修改源代码。

- 误报率 0%（93 条确认，0 条误报）——subagent 验证质量高
- 4 条 ⚠️ 部分正确——修复方案需调整（button L1/L2，thingsboard stop/start 持锁）
- 跨模块维度 E 由本汇总报告统一整合（subagent 各自只看单模块）

**局限性**：
- 静态分析为主，未上机验证
- 部分并发问题（如 TOCTOU）的实际触发概率需上机测试
- destroy 路径的 bug 因 destroy 未被调用而无法通过运行时验证
- LVGL / esp-mqtt / esp-lwlte / iot_button 等第三方库的内部行为基于文档推断，未读源码
