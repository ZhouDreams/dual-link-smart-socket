# BL0942 模块面试准备

分析范围：`main/bl0942/bl0942.h`、`main/bl0942/bl0942.c`，并参考 `docs/agents/classes.md` 中的设计意图。

## 1. 模块一句话职责

`bl0942` 模块负责持有 BL0942 电能计量芯片的 UART、EN GPIO、采样任务和最新测量缓存，通过 `bl0942_read()` 同步读取 23 字节全参数帧，并在内部采样任务中周期性发布 `BL0942_EVENT_MEASUREMENT` 或故障恢复相关的 `BL0942_EVENT_FAULT` 事件。

## 2. 对外 API 总览

Public API 来自 `bl0942.h`，句柄类型是 opaque handle：

| API | 类型 | 作用 |
|-----|------|------|
| `bl0942_create(const bl0942_config_t *config)` | create/init | 应用默认配置、校验参数、分配 `bl0942_t`、创建互斥量和信号量、EN 上下电、安装并配置 UART，成功后返回句柄。 |
| `bl0942_destroy(bl0942_t *me)` | deinit/destroy | 标记销毁、等待正在执行的 public operation 结束、停止采样任务、删除 UART driver、删除信号量和互斥量、释放句柄。 |
| `bl0942_start(bl0942_t *me)` | start/enable | 启动内部采样任务；重复 start 在任务已运行时返回 `ESP_OK`。 |
| `bl0942_stop(bl0942_t *me)` | stop/disable | 请求采样任务退出并等待 `sample_task_done_sema`；任务未启动时返回 `ESP_OK`。 |
| `bl0942_read(bl0942_t *me, bl0942_measurement_t *out)` | process/read | 同步发送 BL0942 packet read 命令，等待 UART 响应，校验帧头和 checksum，解析原始寄存器并更新 latest。 |
| `bl0942_get_latest(bl0942_t *me, bl0942_measurement_t *out)` | get/query | 线程安全地返回最近一次成功测量缓存；暂无数据返回 `ESP_ERR_INVALID_STATE`。 |

推荐调用顺序：

```text
bl0942_create
→ 可选 bl0942_read 做单次同步验证
→ bl0942_start
→ 周期事件 BL0942_EVENT_MEASUREMENT / BL0942_EVENT_FAULT
→ 可选 bl0942_get_latest
→ bl0942_stop
→ bl0942_destroy
```

补充说明：`bl0942_read()` 不要求先 `start()`，因为 UART 在 `create()` 阶段已经安装完成；采样任务运行期间也可以调用 `read()`，但会通过 `mutex` 与任务内读取串行化。

当前项目组合根 `main.c` 传入的 `sample_period_ms` 是 `1000`，所以当前固件实际按 1 Hz 采样；模块内部默认值仍是 `100ms`，只有调用方传 0 或负数时才会使用默认值。

## 3. 生命周期主流程

代码没有显式 state enum，但可以从资源和标志位推断生命周期：

```text
UNINIT
→ bl0942_create 参数合法、资源创建成功、UART 安装配置成功：INITED / STOPPED
→ bl0942_start 创建 sample task 成功：RUNNING
→ 采样成功：RUNNING，更新 latest，发布 MEASUREMENT
→ 连续采样失败达到 fault_threshold：FAULT_HANDLING
→ EN GPIO 硬复位成功：RUNNING
→ 硬复位次数达到 hard_reset_max_attempts：TASK_EXITED / ERROR-LIKE
→ bl0942_stop 成功：INITED / STOPPED
→ bl0942_destroy 成功：UNINIT
```

关键隐含状态字段：

| 状态字段 | 含义 |
|----------|------|
| `initialized` | create 成功后置 true，destroy 删除 UART 前后使用。 |
| `sample_task` | 采样任务句柄，非 NULL 表示任务存在或刚退出待回收。 |
| `sample_task_running` | 采样任务循环条件，stop 和故障超限都会把它置 false。 |
| `stop_in_progress` | 防止并发 stop 或 start/stop 交叠。 |
| `destroying` | destroy 开始后阻止新的 public operation 进入。 |
| `active_ops` | 当前正在执行的 public operation 数量，供 destroy 等待 drain。 |

需要注意：硬复位超限后没有显式 `ERROR` 状态字段，也没有向上提供 `get_status()`。上层只能通过 `BL0942_EVENT_FAULT` 和后续没有测量事件来推断采样任务已经停止。

## 4. 核心数据结构分析

核心结构体是 `struct bl0942`，定义在 `bl0942.c`。

| 类别 | 字段 | 作用 |
|------|------|------|
| 配置 | `bl0942_config_t config` | UART 端口、TX/RX、EN GPIO、波特率、读取超时、采样周期、故障阈值、硬复位次数。 |
| 资源 | `SemaphoreHandle_t mutex` | 串行化 UART 访问和 latest 读写。 |
| 资源 | `SemaphoreHandle_t lifecycle_mutex` | 保护生命周期字段，例如 task handle、running、destroying、active_ops。 |
| 资源 | `TaskHandle_t sample_task` | 周期采样任务句柄。 |
| 资源 | `SemaphoreHandle_t sample_task_done_sema` | 采样任务退出通知，`stop_impl()` 等待它。 |
| 资源 | `SemaphoreHandle_t active_ops_done_sema` | destroy 等待 public operation 清空。 |
| 运行数据 | `bl0942_measurement_t latest` | 最近一次成功读取的原始测量值。 |
| 运行数据 | `bool has_latest` | latest 是否有效。 |
| 运行数据 | `int active_ops` | 正在执行的 public API 数量。 |
| 状态 | `volatile bool sample_task_running` | 采样任务循环标志。 |
| 状态 | `bool stop_in_progress` | stop 正在进行。 |
| 状态 | `bool destroying` | destroy 正在进行，新操作会被拒绝。 |
| 状态 | `bool initialized` | UART 和基础资源已经初始化。 |
| 故障数据 | `uint32_t fault_cycles` | 累计进入 fault_threshold 的次数。 |
| 故障数据 | `int consecutive_failures` | 当前连续读取失败次数。 |
| 故障数据 | `int hard_reset_count` | 当前连续硬复位次数。 |

这个结构体代表模块的“身体部件”：

```text
config          = 先天配置
UART driver     = 和 BL0942 通信的嘴巴和耳朵
EN GPIO         = 故障时硬复位的电源开关
mutex           = UART 和 latest 的访问锁
lifecycle_mutex = 生命周期状态锁
sample_task     = 周期采样心跳
latest          = 最近一次健康读数缓存
fault counters  = 故障检测和恢复记忆
events          = 向 metering_service 输出数据和故障
```

## 5. 资源所有权

| 资源 | 谁创建 | 谁使用 | 谁释放 | 失败回滚 | stop 是否释放 | destroy 是否释放 | 风险点 |
|------|--------|--------|--------|----------|----------------|------------------|--------|
| `bl0942_t *me` | `bl0942_create()` 中 `calloc` | 全部 public API 和采样任务 | `bl0942_destroy()` 中 `free` | create `err:` 释放 | 否 | 是 | destroy 失败时句柄仍归调用者持有。 |
| `mutex` | `xSemaphoreCreateMutex()` | `read/get_latest/hard_reset/destroy` | `vSemaphoreDelete()` | create `err:` 删除 | 否 | 是 | 如果 destroy 中 UART 删除失败，mutex 不释放，供重试 destroy。 |
| `lifecycle_mutex` | `xSemaphoreCreateMutex()` | start/stop/destroy/enter/leave | `vSemaphoreDelete()` | create `err:` 删除 | 否 | 是 | destroy 设置 `destroying=true` 后返回错误，后续普通 API 会被拒绝。 |
| `sample_task_done_sema` | `xSemaphoreCreateBinary()` | stop 等待任务退出，任务退出时 give | `vSemaphoreDelete()` | create `err:` 删除 | 否 | 是 | start 前会 drain 旧信号量，避免误判旧退出事件。 |
| `active_ops_done_sema` | `xSemaphoreCreateBinary()` | destroy 等待 active ops 清空 | `vSemaphoreDelete()` | create `err:` 删除 | 否 | 是 | active operation 机制避免 destroy 与 read/start/stop 并发释放。 |
| UART driver | `bl0942_uart_install()` | `bl0942_read()` 和 hard reset | `bl0942_uart_delete()` | create 中若安装成功则删除 | 否 | 是 | hard reset 中 delete 后 install/configure 失败会让模块继续存在但 UART 不可用。 |
| EN GPIO 配置 | `bl0942_power_cycle()` 中 `gpio_config` | create 和 hard reset | 无显式释放 | 无需释放 | 否 | 否 | GPIO 配置属于 ESP-IDF 全局状态，不做 deinit。 |
| 采样任务 | `xTaskCreate()` | 自身循环调用 `bl0942_read()` | 任务自删 `vTaskDelete(NULL)` | start 失败清空句柄 | stop 等待退出但不删除信号量 | destroy 先 stop 再释放信号量 | 如果任务因故障超限自行退出，`sample_task` 在任务尾部置 NULL。 |
| esp_event base | `ESP_EVENT_DEFINE_BASE(BL0942_EVENT_BASE)` 静态定义 | `esp_event_post()` 发布事件 | 无运行期释放 | 不涉及 | 不涉及 | 不涉及 | 模块只发布，不注册 handler。 |

资源释放对称性结论：

| 场景 | 结论 |
|------|------|
| create 部分失败 | 基本完整回滚，按 UART、信号量、互斥量、内存顺序清理。 |
| stop | 只停止任务，不释放 UART 和同步资源，因此可以重新 start。 |
| destroy | 释放全部模块拥有资源，但如果等待任务或删除 UART 失败，会提前返回并保留句柄。 |
| 重复释放风险 | 正常路径较低，因为 destroy 成功后释放整个对象，调用方不得再使用。 |
| 泄漏风险 | destroy 返回错误时资源有意保留用于重试，不算泄漏；但调用方必须处理失败，否则会长期占用 UART 和信号量。 |
| 悬空指针风险 | destroy 不负责外部串行化句柄生命周期，头文件明确要求调用方在 destroy 成功后不得再使用句柄。 |

## 6. 运行时入口

BL0942 模块内部主要运行时入口只有一个采样任务，没有 timer callback、ISR handler 或协议 receive callback。

| 入口函数 | 类型 | 注册位置 | 触发时机 | 输入事件 | 修改状态 | 主要调用 | 向上输出 |
|----------|------|----------|----------|----------|----------|----------|----------|
| `bl0942_sample_task_entry(void *user_ctx)` | FreeRTOS task | `bl0942_start()` 中 `xTaskCreate()` | start 成功后按 `sample_period_ms` 周期运行 | FreeRTOS 调度和 `sample_task_running` 标志 | 更新 `consecutive_failures`、`hard_reset_count`、`fault_cycles`、`sample_task_running`、`sample_task` | `bl0942_read()`、`bl0942_post_measurement()`、`bl0942_hard_reset()`、`bl0942_post_fault()` | 发布 `BL0942_EVENT_MEASUREMENT` 或 `BL0942_EVENT_FAULT` |

事件消费者不属于本模块，但对讲清事件流很关键：

| 消费者 | 注册位置 | 作用 |
|--------|----------|------|
| `metering_on_bl0942_measurement()` | `metering_service_start()` 注册 `BL0942_EVENT_MEASUREMENT` | 把 BL0942 raw register 换算成业务电参量快照。 |
| `metering_on_bl0942_fault()` | `metering_service_start()` 注册 `BL0942_EVENT_FAULT` | 发布 `valid=false` 的 metering snapshot；如果 BL0942 尝试硬复位，则重置电能增量基线。 |

## 7. 主要方法的主干伪代码

### `bl0942_create()`

1. 检查 `config != NULL`。
2. 应用默认值：波特率、RX buffer、read timeout、sample period、fault threshold、hard reset attempts。
3. 校验 UART、GPIO、device address、baud rate、buffer、timeout、threshold。
4. `calloc` 分配 `bl0942_t`。
5. 创建 `mutex`。
6. 创建 `lifecycle_mutex`。
7. 创建 `sample_task_done_sema`。
8. 创建 `active_ops_done_sema`。
9. 如果配置了 EN GPIO，执行一次 power cycle。
10. 检查 UART driver 是否已被安装，已安装则失败。
11. 安装 UART driver。
12. 配置 UART 参数和 TX/RX 引脚。
13. 初始化 latest、状态标志、故障计数。
14. 返回 handle。

错误路径：

| 失败步骤 | 已创建资源 | 回滚动作 |
|----------|------------|----------|
| 参数检查或配置校验失败 | 无 | 返回 NULL。 |
| `calloc` 失败 | 无 | 返回 NULL。 |
| 创建任一 mutex/semaphore 失败 | 已创建的同步对象和 `me` | 删除已创建同步对象，`free(me)`。 |
| EN power cycle 失败 | `me` 和同步对象 | 删除同步对象，`free(me)`。 |
| UART 已安装 | `me` 和同步对象 | 删除同步对象，`free(me)`。 |
| UART install 失败 | `me` 和同步对象 | 删除同步对象，`free(me)`。 |
| UART configure 失败 | UART driver、`me` 和同步对象 | 删除 UART driver，删除同步对象，`free(me)`。 |

### `bl0942_start()`

1. `bl0942_enter_op()`：拒绝 NULL、destroying 或生命周期资源异常。
2. 检查 `initialized` 和 `sample_task_done_sema`。
3. 获取 `lifecycle_mutex`。
4. 如果正在 destroy 或 stop，返回 `ESP_ERR_INVALID_STATE`。
5. 如果任务已存在且运行中，直接返回 `ESP_OK`。
6. 如果任务句柄存在但退出信号量可取，清理旧 task handle；否则返回状态错误。
7. 重置故障计数。
8. drain `sample_task_done_sema` 中的旧信号。
9. 设置 `sample_task_running=true`。
10. `xTaskCreate(bl0942_sample_task_entry, ...)`。
11. 创建失败则恢复 `sample_task_running=false` 和 `sample_task=NULL`，返回 `ESP_ERR_NO_MEM`。
12. 释放 lifecycle lock，`bl0942_leave_op()`。

### `bl0942_stop()` / `bl0942_stop_impl()`

1. public `bl0942_stop()` 先 `enter_op()`。
2. `stop_impl()` 校验句柄、initialized、semaphore、lifecycle mutex。
3. 获取 `lifecycle_mutex`。
4. 如果 `sample_task == NULL`，设置 `sample_task_running=false`，返回 `ESP_OK`。
5. 如果当前任务就是采样任务，禁止自 stop，返回 `ESP_ERR_INVALID_STATE`。
6. 如果已有 stop 进行中，返回 `ESP_ERR_INVALID_STATE`。
7. 设置 `sample_task_running=false` 和 `stop_in_progress=true`。
8. 释放 lifecycle lock。
9. 等待 `sample_task_done_sema`，超时时间为 `read_timeout + sample_period + 3500ms`。
10. 重新获取 lifecycle lock，清除 `stop_in_progress`。
11. 等待成功则 `sample_task=NULL`，返回 `ESP_OK`；超时返回 `ESP_ERR_TIMEOUT`。
12. public `bl0942_stop()` 最后 `leave_op()`。

错误路径重点：stop 超时不会释放 task 资源，也不会删除 UART，调用者可以重试 stop 或 destroy。

### `bl0942_destroy()`

1. `me == NULL` 时直接返回 `ESP_OK`。
2. 获取 `lifecycle_mutex`，设置 `destroying=true`，释放锁。
3. `bl0942_wait_active_ops_drained()` 等待并发 public API 退出。
4. 调用 `bl0942_stop_impl()` 停止采样任务。
5. 获取 `mutex`。
6. 删除 UART driver。
7. 设置 `initialized=false`。
8. 删除 `sample_task_done_sema`、`active_ops_done_sema`、`lifecycle_mutex`、`mutex`。
9. `free(me)`。

错误路径：

| 失败点 | 行为 |
|--------|------|
| 等待 active ops 失败 | 返回错误，句柄保留，但 `destroying=true`。 |
| stop task 失败 | 返回错误，句柄保留，可重试 destroy。 |
| 获取 `mutex` 失败 | 返回 `ESP_ERR_TIMEOUT` 或状态错误，资源保留。 |
| UART delete 失败 | 释放 mutex 后返回错误，资源保留，可重试 destroy。 |

### `bl0942_read()`

1. `enter_op()`。
2. 检查 `out != NULL`、`initialized`、`mutex`。
3. 获取 `mutex`，串行化 UART。
4. 构建两字节读取命令。
5. 如果 `destroying` 或 UART driver 未安装，返回 `ESP_ERR_INVALID_STATE`。
6. `uart_flush_input()` 清空旧数据。
7. `uart_write_bytes()` 发送命令。
8. `uart_wait_tx_done()` 等待发送完成。
9. `uart_read_bytes()` 读取 23 字节响应。
10. 检查帧头 `0x55`。
11. 计算并校验 checksum。
12. 解析原始寄存器，填充 timestamp 和 `valid=true`。
13. 更新 `latest` 和 `has_latest`。
14. 返回测量结果。
15. 释放 `mutex`，`leave_op()`。

错误路径：

| 失败点 | 返回 | 影响 |
|--------|------|------|
| 参数或状态错误 | `ESP_ERR_INVALID_ARG` / `ESP_ERR_INVALID_STATE` | 不访问 UART。 |
| 获取锁失败 | `ESP_ERR_TIMEOUT` | 不访问 UART。 |
| flush/write/wait 失败 | driver 错误或 `ESP_FAIL` | 不更新 latest。 |
| 读取长度不足 | `ESP_ERR_TIMEOUT` | 采样任务会计入连续失败。 |
| 帧头或 checksum 错误 | `ESP_ERR_INVALID_RESPONSE` | 采样任务会计入连续失败。 |

### `bl0942_get_latest()`

1. `enter_op()`。
2. 检查 `out != NULL`、`initialized`、`mutex`。
3. 获取 `mutex`。
4. 如果 destroying 或 `has_latest=false`，返回 `ESP_ERR_INVALID_STATE`。
5. 拷贝 latest 到 out。
6. 释放 `mutex`，`leave_op()`。

### `bl0942_sample_task_entry()`

1. 获取 `me`，记录 `last_wake`。
2. 当 `sample_task_running` 为 true 时循环。
3. 调用 public `bl0942_read()` 同步读取。
4. 成功：清零连续失败和硬复位计数，发布 `BL0942_EVENT_MEASUREMENT`。
5. 失败：`consecutive_failures++`。
6. 如果连续失败达到 `fault_threshold`：
7. `fault_cycles++`。
8. 如果 EN GPIO 可用且 `hard_reset_count < hard_reset_max_attempts`，执行 `bl0942_hard_reset()`。
9. 发布 `BL0942_EVENT_FAULT`，payload 包含连续失败数、fault cycles、是否尝试 reset、last error。
10. 清零 `consecutive_failures`。
11. 如果硬复位次数达到上限且上限大于 0，停止采样循环。
12. `vTaskDelayUntil()` 等待下一个采样周期。
13. 退出时设置 `sample_task_running=false`，`sample_task=NULL`，give `sample_task_done_sema`，`vTaskDelete(NULL)`。

| `bl0942_uart_delete` | C 清理/回滚函数 | 是 | 删除 UART driver，支持未安装时返回 OK。 |
| `bl0942_hard_reset` | C 清理/回滚函数 | 是 | 运行时软恢复：EN power cycle 后重建 UART。 |
| `bl0942_enter_op` | A 阶段函数 | 是 | public API 入口计数，阻止 destroy 后新操作进入。 |
| `bl0942_leave_op` | A 阶段函数 | 是 | public API 退出计数，为 destroy drain 发信号。 |
| `bl0942_wait_active_ops_drained` | C 清理/回滚函数 | 是 | destroy 等待并发 public API 清空。 |
| `bl0942_stop_impl` | C 清理/回滚函数 | 是 | stop/destroy 共用的任务停止实现。 |
| `bl0942_sample_task_entry` | D task/callback 入口 | 是 | 周期采样、故障计数、硬复位和事件发布的运行时主循环。 |

可能过度封装、可考虑合并回主函数的 helper：

| 函数 | 判断 |
|------|------|
| `bl0942_build_read_cmd` + `bl0942_build_packet_read_cmd` | 两者都很短，如果没有协议单测需求，可以合并为一个 helper 或直接放在 `read()` 中。 |
| `bl0942_uart_install` | 只是 `uart_driver_install()` 的一行包装，除非为了统一命名阶段，否则可以内联。 |
| `bl0942_is_supported_baud_rate` | 当前 switch 很短，保留可读性尚可，但不是主干。 |

不建议合并的 helper：`bl0942_stop_impl()`、`bl0942_hard_reset()`、`bl0942_enter_op()`、`bl0942_leave_op()`、`bl0942_wait_active_ops_drained()`。这些函数承载生命周期并发和故障恢复语义，合并后反而更难审核资源释放。

## 9. 错误码分类

| 分类 | 错误码 | 发生条件 | 调用者用错还是运行时故障 | 是否触发清理/重试/恢复 | 返回给谁 |
|------|--------|----------|--------------------------|--------------------------|----------|
| 参数错误 | `ESP_ERR_INVALID_ARG` | `config == NULL`、GPIO/UART/address/baud/buffer/timeout 不合法、`out == NULL` | 多数是调用者配置错误 | create 返回 NULL；read/get_latest/start/stop 返回错误，不重试 | 直接返回 public API 调用者 |
| 状态错误 | `ESP_ERR_INVALID_STATE` | 未 initialized、destroying、UART 未安装、stop/start 交叠、无 latest、采样任务自 stop | 多数是调用顺序或生命周期并发问题，也可能是 hard reset 后 UART 不可用 | destroy/start/stop 不自动修复；采样任务中 read 返回该错误会计入故障 | public API 调用者或采样任务 fault 逻辑 |
| 资源错误 | `ESP_ERR_NO_MEM` | `xTaskCreate()` 失败；create 中 `calloc` 或 semaphore/mutex 创建失败表现为返回 NULL | 运行时资源不足 | create 回滚已创建资源；start 创建任务失败会清理 task flag | create 调用者或 start 调用者 |
| 超时错误 | `ESP_ERR_TIMEOUT` | 获取 mutex/lifecycle_mutex 失败、UART 读不到完整帧、TX wait 超时、stop 等任务退出超时 | 可能是系统调度问题，也可能是外设无响应 | 采样任务中 read timeout 会累计失败并触发 hard reset；stop timeout 不强杀任务 | public API 调用者或 fault event payload |
| 协议响应错误 | `ESP_ERR_INVALID_RESPONSE` | 帧头不是 `0x55` 或 checksum 不匹配 | 外设通信异常、噪声、波特率/接线问题 | 采样任务累计失败，达到阈值后发布 fault 并尝试 hard reset | public API 调用者或 fault event payload |
| 通用通信失败 | `ESP_FAIL` | `uart_write_bytes()` 写入长度不等于命令长度 | UART driver 或底层通信异常 | 采样任务累计失败，达到阈值后恢复 | public API 调用者或 fault event payload |
| ESP-IDF driver 错误 | UART/GPIO API 原始返回值 | `gpio_config`、`gpio_set_level`、`uart_driver_install/delete`、`uart_param_config`、`uart_set_pin` 失败 | 外设资源冲突或 driver 错误 | create 阶段回滚；hard reset 阶段可能留下 UART 不可用状态 | public API 调用者或 fault event payload |
| 严重不可恢复错误 | `ESP_ERROR_CHECK` | 本模块没有使用 | 不适用 | 不适用 | 不适用 |

错误处理风格结论：模块内部没有自定义错误码，也没有 `ESP_ERROR_CHECK` 直接 abort。普通错误通过 `esp_err_t` 返回，运行时采样错误通过 fault counter 转换为 `BL0942_EVENT_FAULT`。

## 10. 故障恢复机制

### Level 1：初始化失败回滚

| 失败阶段 | 已申请资源 | 回滚方式 |
|----------|------------|----------|
| 配置非法 | 无 | 返回 NULL。 |
| 分配 `me` 后同步资源创建失败 | `me` 和部分 semaphore/mutex | 删除已创建 semaphore/mutex，`free(me)`。 |
| EN power cycle 失败 | `me` 和同步资源 | 删除同步资源，`free(me)`。 |
| UART 已安装或安装失败 | `me` 和同步资源 | 删除同步资源，`free(me)`。 |
| UART configure 失败 | UART driver、`me`、同步资源 | `bl0942_uart_delete()`，删除同步资源，`free(me)`。 |

Level 1 做得比较完整，符合 create 失败不返回半初始化对象的原则。

### Level 2：运行时软恢复

运行时恢复由 `bl0942_sample_task_entry()` 驱动：

```text
bl0942_read 失败
→ consecutive_failures++
→ 达到 fault_threshold
→ fault_cycles++
→ 如果 EN GPIO 可用且 hard_reset_count 未超限
→ bl0942_hard_reset
   → EN power cycle
   → uart_driver_delete
   → uart_driver_install
   → uart_param_config + uart_set_pin
→ 发布 BL0942_EVENT_FAULT
→ consecutive_failures 清零
→ 未超限则继续采样
```

软恢复能力：

| 能力 | 是否存在 | 说明 |
|------|----------|------|
| retry | 有 | 每次读取失败只计数，不立即 reset；达到阈值后再恢复。 |
| reconnect | 不适用 | BL0942 是 UART 外设，没有网络连接。 |
| reset driver | 有 | hard reset 会删除并重装 UART driver。 |
| 重新注册事件 | 不需要 | 本模块只 post event，不注册 handler。 |
| 上报错误事件 | 有 | `BL0942_EVENT_FAULT` 携带 last error、fault cycles、reset attempted。 |

### Level 3：模块级重启

模块内部没有完整的 stop → destroy → create → start 级别重启流程，也不会调用 `esp_restart()`。

| 项 | 结论 |
|----|------|
| stop → deinit → init → start | 不在模块内部做，需要 app_controller 或上层策略决定。 |
| ERROR 状态 | 无显式字段。硬复位超限后采样任务退出，表现为不再发布测量事件。 |
| 最大失败次数 | 有，来自 `hard_reset_max_attempts`。注意这是硬复位次数上限，不是 read 失败总次数。 |
| esp_restart | 没有。 |

改进建议：如果后续需要更清晰的故障恢复，可以增加 `bl0942_get_status()` 或显式 `state` 字段，让上层区分 STOPPED、RUNNING、FAULT_RECOVERING、ERROR。

## 11. 事件流 / 数据流

BL0942 是 task + polling + esp_event 输出的混合型模块：

```text
app_controller
→ bl0942_start
→ xTaskCreate(bl0942_sample_task_entry)
→ sample task 周期循环
→ bl0942_read
→ uart_flush_input
→ uart_write_bytes(packet read command)
→ uart_wait_tx_done
→ uart_read_bytes(23-byte frame)
→ frame header + checksum 校验
→ parse raw registers
→ 更新 latest
→ esp_event_post(BL0942_EVENT_MEASUREMENT)
→ metering_service handler
→ 工程量换算和业务快照
```

故障流：

```text
UART timeout / invalid header / checksum mismatch / driver error
→ bl0942_read 返回错误
→ sample task consecutive_failures++
→ 达到 fault_threshold
→ 可选 EN hard reset + UART driver 重建
→ esp_event_post(BL0942_EVENT_FAULT)
→ metering_service fault handler
→ 发布 valid=false 的 metering snapshot
→ safety / display / app 层感知数据失效
```

模块类型判断：

| 类型 | 是否符合 | 说明 |
|------|----------|------|
| 主动轮询型 | 是 | sample task 周期主动发 UART 读命令。 |
| 事件驱动型 | 部分是 | 输出通过 esp_event 广播。 |
| callback 驱动型 | 否 | 本模块不接收用户 callback。 |
| task + queue 驱动型 | task 是，queue 否 | 有 FreeRTOS task，但没有 queue。 |
| 混合型 | 是 | 同步 UART read + task polling + esp_event output。 |

## 12. 最终总结：模块卡片

模块名：`bl0942`

一句话职责：封装 BL0942 的 UART 通信、EN 硬复位、周期采样任务、最新原始测量缓存和测量/故障事件发布。

核心 API：

| API 类别 | 函数 |
|----------|------|
| init | `bl0942_create()` |
| start | `bl0942_start()` |
| stop | `bl0942_stop()` |
| send/process | `bl0942_read()` |
| query | `bl0942_get_latest()` |
| deinit | `bl0942_destroy()` |

生命周期：

```text
UNINIT → INITED/STOPPED → RUNNING → FAULT_RECOVERING → RUNNING 或 TASK_EXITED/ERROR-LIKE → STOPPED → UNINIT
```

核心状态：`initialized`、`sample_task_running`、`sample_task`、`stop_in_progress`、`destroying`、`active_ops`、`has_latest`、`consecutive_failures`、`hard_reset_count`、`fault_cycles`。

拥有资源：`bl0942_t` 内存、UART driver、EN GPIO 配置、采样任务、两个 mutex、两个 binary semaphore、latest cache、esp_event 输出能力。

运行时入口：`bl0942_sample_task_entry()`，由 `bl0942_start()` 通过 `xTaskCreate()` 注册。

正常主流程：start 后采样任务按周期调用 `bl0942_read()`，通过 UART 读取 23 字节帧，校验帧头和 checksum，解析 raw register，更新 latest，并发布 `BL0942_EVENT_MEASUREMENT`。

主要错误类型：参数配置错误、生命周期状态错误、内存/任务创建失败、UART 超时、协议响应错误、UART/GPIO driver 错误。

故障恢复策略：读取失败先累计计数，达到阈值后发布 fault；如果配置 EN GPIO 且硬复位次数未超限，则 EN 上下电并重建 UART driver；硬复位超限后停止采样任务，不执行系统重启。

我面试时可以这样讲：

> BL0942 这个模块我把它设计成驱动适配层，核心是一个 opaque handle，里面持有 UART、EN GPIO、采样任务、同步锁和最新测量缓存。对上层暴露 create/start/read/get_latest/stop/destroy 这组生命周期 API，start 后由内部 FreeRTOS 任务主动轮询芯片，读取 23 字节全参数帧并做帧头和 checksum 校验，成功后通过 esp_event 发布原始测量事件，让 metering_service 再做工程量换算。故障恢复上不是读失败就重启，而是用连续失败阈值过滤瞬时错误，达到阈值后发布故障事件，并在 EN GPIO 可用时做硬复位和 UART driver 重建。资源管理上 create 失败会逆序回滚，stop 只停任务保留 UART，destroy 才释放 UART、信号量、互斥量和对象内存；同时用 lifecycle mutex 和 active_ops 计数避免 destroy 与 read/start/stop 并发释放。这个模块的不足是没有显式 state enum 或 get_status，硬复位超限后只能通过 fault 事件和测量事件停止来推断异常状态，后续可以补一个状态查询接口增强可观测性。
