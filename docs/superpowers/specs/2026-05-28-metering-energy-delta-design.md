# Metering Energy Delta Telemetry Design

## Goal

Restore the energy telemetry contract to cloud-accumulated energy: the device uploads only the interval energy increment, and ThingsBoard computes total energy. The metering layer owns BL0942 CF-counter interpretation and energy conversion; the app layer only publishes telemetry and confirms successful delivery.

## Current Behavior

`metering_service` currently treats BL0942 `cf_cnt_raw` as a source for local accumulated energy. The first valid sample initializes the local baseline. Each later valid sample computes a 24-bit wrap-safe CF count delta, multiplies it by `METERING_PULSE_ENERGY_NWH`, accumulates into `total_energy_nwh`, converts to Wh, and exposes that value as `snapshot.total_energy` and telemetry `totalEnergy`.

This is not the desired contract. It makes the device the owner of total energy, and the telemetry field is a cumulative Wh value rather than a cloud-accumulated interval increment.

## Required Behavior

- Sample BL0942 at 1 Hz.
- On each successful BL0942 sample, emit one metering snapshot immediately.
- Do not aggregate a 10 Hz metering window for normal telemetry.
- Keep BL0942 CF-counter conversion inside `metering_service`.
- Compute `energyDelta` from the current CF counter and the last cloud-confirmed CF baseline.
- Expose `energyDelta` in mWh to telemetry.
- Only advance the confirmed energy baseline after ThingsBoard publish returns `ESP_OK`.
- If publish fails, release the pending delta without advancing the confirmed baseline so the next `energyDelta` includes the not-yet-uploaded energy.
- Include `frequency` in telemetry.
- Make `app_controller` and `metering_service` logs use the same measurement fields, order, and units.

## BL0942 CF Counter Semantics

BL0942 `cf_cnt_raw` is a 24-bit cumulative energy pulse counter. It increases as energy is consumed and wraps at 24 bits. Energy delta must therefore be computed with wrap-safe unsigned arithmetic:

```c
delta = (current_cf_cnt - baseline_cf_cnt) & 0x00FFFFFF;
```

The project coefficient is `62297938 nWh/pulse`. The interval energy is:

```c
delta_nwh = delta * 62297938 + residual_nwh;
energyDelta_mWh = floor(delta_nwh / 1000) / 1000.0;
next_residual_nwh = delta_nwh % 1000;
```

The residual is retained only after a successful publish confirmation, matching the old project's precision-preserving behavior.

## Architecture

### Metering Service

`metering_service` remains the owner of electrical measurement semantics. It converts BL0942 raw registers into engineering units and converts CF count differences into energy increments.

The snapshot will carry:

- `voltage` in V
- `current` in A
- `power` in W
- `energy_delta` in mWh
- `frequency` in Hz
- `timestamp_us`
- `valid`
- `energy_delta_token`, an opaque `uint32_t` confirmation token generated by `metering_service`

The app layer must not calculate CF deltas or know the pulse-energy coefficient.

`metering_service` will expose this confirmation API:

```c
esp_err_t metering_service_confirm_energy_delta(
    metering_service_t *me,
    const metering_snapshot_t *snapshot);
```

The confirm API advances the last-confirmed CF baseline and residual only if `snapshot->energy_delta_token` matches the current pending token. This prevents app-layer retries or stale snapshot confirmations from corrupting the baseline. The token is opaque to `app_controller`; it must only pass the snapshot back after a successful publish.

`metering_service` will also expose a failed-publish release API:

```c
esp_err_t metering_service_discard_energy_delta(
    metering_service_t *me,
    const metering_snapshot_t *snapshot);
```

The discard API clears the pending token only if it matches the snapshot. It does not advance the confirmed CF baseline or residual, so the next prepared `energyDelta` covers the failed interval.

### App Controller

`app_controller` builds telemetry from the metering snapshot. It forwards `energy_delta` and `frequency` to ThingsBoard. After `thingsboard_client_publish_telemetry()` returns `ESP_OK`, it calls the metering confirmation API for that snapshot. If publish returns an error, it calls the metering discard API for that snapshot.

`app_controller` must not store `last_successful_cf_cnt_raw`, compute wrap-safe CF deltas, or apply `METERING_PULSE_ENERGY_NWH`.

### ThingsBoard Telemetry

Telemetry will replace cumulative `totalEnergy` with interval `energyDelta`:

```json
{
  "voltage": 230.00,
  "current": 0.123,
  "power": 12.34,
  "energyDelta": 0.123,
  "frequency": 50.00,
  "relayOn": true,
  "activeLink": "wifi",
  "safetyLevel": 0,
  "valid": true
}
```

`energyDelta` unit is mWh. `frequency` unit is Hz.

## Data Flow

1. BL0942 produces a valid 1 Hz measurement.
2. `metering_service` converts raw registers to V/A/W/Hz.
3. `metering_service` computes `energyDelta` from current CF count and last confirmed baseline.
4. `metering_service` emits one snapshot.
5. `app_controller` publishes the snapshot as telemetry.
6. On publish success, `app_controller` calls `metering_service_confirm_energy_delta()`.
7. `metering_service` advances the confirmed CF baseline and residual.
8. On publish failure, `app_controller` calls discard; `metering_service` clears pending state without advancing the baseline, and the next snapshot includes the missed energy interval.

## Logging

Both metering and publish-success logs will use this measurement order and unit set:

```text
V=%.2fV I=%.3fA P=%.2fW E=%.3fmWh F=%.2fHz valid=%d
```

`app_controller` may append publish sequence and link information, but `E` must mean `energyDelta` in mWh in both modules.

## Error Handling

- Invalid BL0942 samples do not emit valid telemetry snapshots.
- A first valid CF count establishes the initial baseline and produces `energyDelta=0.000`.
- CF counter wrap is handled with 24-bit masked subtraction.
- Publish failure discards only the pending token and does not advance the energy confirmation baseline.
- Stale or mismatched confirmations return an error and do not change energy state.
- Hard reset or meter reset clears pending CF baseline state so the next valid sample re-baselines.
