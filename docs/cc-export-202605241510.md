  Key Rationale

  - relay and button before bl0942 — They're trivially simple (one GPIO each)
  and let you validate the opaque handle + event pattern before tackling UART
    protocol parsing.
  - metering_service before safety_guard — Safety needs metering snapshots as
  input.
  - wifi_link before lte_link — Wi-Fi is simpler to develop and test (no
  external hardware), and network_manager needs at least one working link to be
    testable.
  - network_manager before thingsboard_client — ThingsBoard needs the unified
  publish/subscribe interface.
  - app_controller last — It wires everything together. Its API depends on every
   other module being finalized.

  The key question before starting Batch 1 is: do you want to write classes.md
  for all 14 modules upfront (like esp-lwlte did), or define classes per batch
  as we go? Given that esp-lwlte's classes.md was written as a complete
  reference before any code, I'd recommend the same approach for Smart_Socket —
  it forces you to think through every module's API, visibility, and OOP role
  before implementation begins.

      685 +### 5.7 关键设计决策
      686 +
      687 +- `bl0942_start()` 启动内部采样任务，周期由 `sample_period_ms`
          + 配置（默认 100ms = 10Hz）
      688 +- `bl0942_read()` 是同步阻塞接口，与内部采样任务通过 mutex 串
          +行化，供启动前芯片验证或临时单次读取
      689 +- `bl0942_get_latest()` 返回缓存的最新测量值，无锁读取
      690 +- 内部故障检测：连续失败达 `fault_threshold` → 发布 `BL0942_EV
          +ENT_FAULT` → 自动 EN-pin 硬复位 → 硬复位超 `hard_reset_max_att
          +empts` 则停止采样任务
      691 +- UART 配置只用单一波特率（`baud_rate`），不做启动波特率→运行
          +波特率切换
      692 +- 原始寄存器值不在此层转换工程量，换算逻辑属于 `metering_servi
          +ce`

      696 +## 6. Network Link（网络链路基类接口）
      697 +
      698 +Network Link 是网络抽象层的核心，定义 Wi-Fi/LTE 双模链路的统一
          +基类接口，通过 ops 表实现继承与多态。`network_manager` 只依赖

      699 +
      700 +### 6.1 类总览
      701 +
      702 +| 类 | 可见性 | 被谁使用 | OOP 角色 | 说明 |
      703 +|----|--------|---------|---------|------|
      704 +| `network_link_t` | 用户 API (opaque) | network_manager | 句
          +柄/基类 | 所有链路的统一句柄，struct 定义在 .c 或 _priv.h |

      707 +以下类型已在 Batch 1 的 `network_types.h` 中定义，`network_lin

      711 +
      712 +**所属层**：网络抽象层
      713 +**可见性**：内部 — 定义在 `network_link_priv.h`，只给子类实现

      714 +**OOP 角色**：虚函数表
      715 +
      716 +```c
      717 +typedef struct {
      718 +    esp_err_t (*destroy)(network_link_t *me);
      719 +    esp_err_t (*start)(network_link_t *me);
      720 +    esp_err_t (*stop)(network_link_t *me);
      721 +    esp_err_t (*get_status)(network_link_t *me, network_link_s
          +tatus_t *out);
      722 +    esp_err_t (*publish)(network_link_t *me, const network_pub
          +lish_request_t *req);
      723 +    esp_err_t (*subscribe)(network_link_t *me, const char *top
          +ic,
      724 +                            network_mqtt_qos_t qos);
      725 +    esp_err_t (*unsubscribe)(network_link_t *me, const char *t
          +opic);
      726 +    esp_err_t (*register_rx_cb)(network_link_t *me, network_rx
          +_cb_t cb,
      727 +                                 void *ctx);

      730 +
      731 +**方法语义**：
      732 +
      733 +| 方法 | 语义 |
      734 +|------|------|
      735 +| `destroy` | 释放子类所有资源，停止任务/断开连接/注销回调，最
          +后释放自身内存 |
      736 +| `start` | 启动物理连接 + MQTT 建链，异步推进，状态通过 `get_
          +status` 观察 |
      737 +| `stop` | 断开 MQTT + 物理连接，保留已分配资源和配置，可重新
          +`start` |
      738 +| `get_status` | 返回当前链路快照（link_type、state、link_up、
          +mqtt_ready） |
      739 +| `publish` | 通过当前链路的 MQTT 连接发布消息 |
      740 +| `subscribe` | 在当前链路上注册主题订阅，支持重连后重放 |
      741 +| `unsubscribe` | 撤销订阅 |
      742 +| `register_rx_cb` | 注册下行消息回调，同一时刻只保留一个回调
          +|

      812 +    .get_status     = wifi_link_get_status_impl,
      813 +    .publish        = wifi_link_publish_impl,
      814 +    .subscribe      = wifi_link_subscribe_impl,
      815 +    .unsubscribe    = wifi_link_unsubscribe_impl,

          +fig)

      852 +### 6.6 关键设计决策
      853 +
      854 +- 基类结构极简 — 只含 `ops` 和 `type`，不预设状态、锁、回调等
          +字段，每个子类管理自己的内部状态
      855 +- `network_link_ops_t` 定义在 `network_link_priv.h`，不暴露给
          +`network_manager` — manager 只通过包装函数调用，不直接访问 ops
          + 表
      856 +- 包装函数统一做参数校验（NULL 检查、ops 方法存在性检查），再
          +委托给 ops 方法，保证上层调用安全
      857 +- `network_link_get_type()` 直接读 `me->type`，不经过 ops 表（
          +类型查询不需要多态）
      858 +- 子类必须把 `network_link_t base` 放第一个字段，保证 `&self->
          +base == (network_link_t *)self`
      859 +- `destroy` 进入 ops 表，`network_manager` 切换链路或销毁时可
          +以统一释放，不关心子类类型
      860 +- 子类工厂函数（`wifi_link_create` / `lte_link_create`）返回 `
          +network_link_t *`，不暴露子类具体类型

❯ git rm是干嘛的

⏺ git rm --cached 是从 git 暂存区移除文件，但不删除磁盘上的实际文件。因为之前的
  git add -A 已经把 reference/ 下的 3000+
  文件暂存了，需要先用这个命令把它们从暂存区去掉，再配合 .gitignore 让 git
  永久忽略它们。

  更简单的做法是直接重置重新来：

  git reset HEAD .           # 清空暂存区
  git add -A                 # 重新暂存（这次 reference/ 会被 .gitignore 忽略）
  git commit -m "..."        # 提交
